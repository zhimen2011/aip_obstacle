# 数据流说明

本文档描述 **第一阶段 MVP** 的数据流。仅涵盖已决定实现的部分，未实现（OCR 等）会显式标注「预留」。

数据来源限定为 **中国民航 AIP**。

---

## 总体数据流

```text
用户通过命令行或 GUI 指定输入文件
  -> [parsers] 读取文件（PDF 文本层 / TXT）
  -> [parsers] 抽取原始文本 + 页码 + 机场 ICAO（来自页眉或章节标题）
  -> [services] 识别「疑似障碍物行」
  -> [services] 解析字段（编号 / 名称 / 方位 / 距离 / 经纬度 / 高度）
  -> [services] 字段标准化（单位、坐标）
  -> [ui]       生成本次确认表：成功记录 + 解析失败占位行
  -> [ui]       用户人工修改、补录待补录行、必要时新增行
  -> [storage]  用户点击“最终确认”后，当前确认表覆盖写入 SQLite.obstacles
  -> [exporters] 最终确认后导出 XLSX
  -> [utils.logging] 写处理日志
```

一个文件的处理**不会**因为某一行解析失败而中断。GUI 中解析失败的行会进入主确认表占位，用户补录后再最终确认；命令行流程仍可把失败行写入 `parse_failures`。

缺经纬度不作为人工审核、颜色标注或 `confidence_score` 的判断依据。经纬度为空的记录仍保留在普通障碍物数据中；只有导出 GeoJSON 时，因为 GeoJSON 必须有坐标，才跳过该 feature。

GUI 表格颜色只表达当前处理状态：红色表示高风险，必须人工复核；黄色表示存在风险，需要关注；蓝色表示已经经过人工修改。人工修改状态优先于自动风险颜色。

GUI 不再在解析完成时自动写库，也不会因为文件 hash 已存在而读取旧数据覆盖本次解析结果。只有用户点击“最终确认”后，才会用当前确认表覆盖同一文件 hash 对应的旧数据库记录。XLSX 导出只能读取本次最终确认后的数据。

## 数据输入

MVP 接受的输入：

| 类型 | 说明 | 备注 |
|------|------|------|
| 文本型 PDF | 中国 AIP 发布的 PDF，能直接复制文字 | 主要输入 |
| TXT | 已导出的纯文本 | 测试 / 调试 / 兜底 |
| 扫描型 PDF | 无文本层 | **本期不支持**，检测到即跳过 |
| 图片 / Word / Excel | — | **本期不支持** |
| 境外 AIP | — | **本期不支持** |

预留接口（不实现）：

- `parsers/ocr_parser.py`：OCR 通道占位
- 在线下载 AIP：不做

---

## 数据校验

在数据进入处理管线前，做最小校验：

- 文件是否存在、是否可读
- 扩展名是否为 `.pdf` 或 `.txt`
- PDF 是否有文本层（无则提示并跳过）
- PDF 是否加密（加密则提示并跳过）
- 输出目录是否可写（不存在自动创建）

字段级校验发生在 `services` 阶段，失败时进入 `parse_failures`，不抛出终止整个流程。

## 数据处理

按阶段拆分，每一阶段对应一个模块：

### 阶段 1：文本抽取（parsers）

- 输入：PDF / TXT 路径
- 输出：`List[TextBlock]`，每个块至少包含：
  - `page`：页码（TXT 固定为 1，或按换页符切分）
  - `text`：该页 / 该块的原始文本
  - `airport_icao`：从页眉或章节标题识别出的机场 ICAO 四字码（识别不到则为 `UNKNOWN`）
- 关键点：**保留页码与机场归属**，便于后续溯源

### 阶段 2：识别疑似障碍物行（services）

- 输入：`List[TextBlock]`
- 输出：`List[CandidateRow]`（`raw_text` + `source_page` + `airport_icao`）
- MVP 宽松识别规则（以中国 AIP 常见的障碍物表为参照）：
  - 行首或靠前位置出现「编号」模式（数字、编号字样）
  - 行内出现「高度相关关键词 / 数字 + 单位」或「经纬度」或「方位 + 距离」任一组合
  - 过滤明显的表头、页眉、页脚

### 阶段 3：字段解析与标准化（services）

- 输入：`CandidateRow`
- 输出：`ObstacleRecord` 或 `ParseFailure`
- 标准化规则：
  - **经纬度**：度分秒（中国 AIP 常见 `DDMMSS.ssN` / `DDDMMSS.ssE` 格式）→ 十进制度（WGS84）
  - **方位 bearing_deg**：统一为 0–360 十进制度；若原文标注为磁方位（MAG），本期不换算，记录到警告
  - **距离 distance_m**：统一换算为米；支持 km / m / NM
  - **高度 / 海拔**：ft → m；保留原始单位字段
  - **编号 / 名称**：字符串原样保留，去掉两端空白
- 成功条件（进 obstacles）：
  - `airport_icao` + `obstacle_id` + `name` 三个必填都有
  - 且 `(bearing_deg + distance_m)` 至少一对完整 或 `(latitude + longitude)` 至少一对完整
- 否则进 parse_failures，并在 `reason` 中写明缺失项

### 阶段 4：落库（storage）

- SQLite，表结构（MVP 版，后续可加字段不破坏兼容）：

```text
source_files
  id INTEGER PK
  file_path TEXT
  file_hash TEXT        -- sha256，用于去重
  imported_at TEXT      -- ISO8601
  total_candidates INT
  total_success INT
  total_failed INT

obstacles
  id INTEGER PK
  source_file_id INT FK
  airport_icao TEXT           -- 如 ZBAA；未识别则 UNKNOWN
  obstacle_id TEXT            -- AIP 原始编号
  name TEXT                   -- 障碍物名称
  bearing_deg REAL            -- 方位（真北，度），可空
  distance_m REAL             -- 距离（米），可空
  latitude REAL               -- 纬度（十进制度 WGS84），可空
  longitude REAL              -- 经度（十进制度 WGS84），可空
  elevation_m REAL            -- 海拔高（米），可空
  height_m REAL               -- 相对地面高（米），可空
  unit_distance_original TEXT -- 原文距离单位（m/km/NM）
  unit_height_original TEXT   -- 原文高度单位（ft/m）
  is_user_modified INT        -- 0/1，是否经过人工修改
  edited_at TEXT              -- 人工修改时间，ISO8601，可空
  source_page INT
  raw_text TEXT
  UNIQUE(airport_icao, obstacle_id, raw_text)

parse_failures
  id INTEGER PK
  source_file_id INT FK
  airport_icao TEXT           -- 尽力识别，可能是 UNKNOWN
  source_page INT
  raw_text TEXT
  reason TEXT                 -- 哪一步、哪个字段失败
```

### 阶段 5：导出（exporters）

- **CSV** `obstacles.csv`：所有记录，列与数据库字段一致，缺字段留空，包含 `is_user_modified` / `edited_at`
- **JSON** `obstacles.json`：所有记录，数组形式，缺字段为 `null`
- **GeoJSON** `obstacles.geojson`：**仅 `latitude` 和 `longitude` 都非空的记录**生成 feature；`geometry` 为 `Point[lon, lat]`，其他字段放到 `properties`；缺坐标的条目跳过并计数
- **XLSX** `obstacles.xlsx`：GUI 中只能在“最终确认”后导出；单表导出本次确认数据，保留确认表顺序，序号列使用 AIP 原始 `obstacle_id`；不按距离重新排序，也不根据距离拆成左右栏；增加“是否人工修改”列
- 失败记录另行导出：`parse_failures.csv`（默认导出）

导出时不再额外生成“缺经纬度审核清单”。如果用户在 GUI 中改过字段或补录了失败行，最终确认后导出使用本次确认值；未修改字段继续使用自动解析值。

## 数据输出

MVP 的输出清单：

```text
<out_dir>/
├── aip_obstacle.sqlite      # 主数据库
├── obstacles.csv            # 全量
├── obstacles.json           # 全量
├── obstacles.geojson        # 仅含经纬度的条目
├── obstacles.xlsx           # Excel 单表，保留原始顺序和 AIP 原始编号
├── parse_failures.csv
└── logs/
    └── run_YYYYMMDD_HHMMSS.log
```

## 数据存储

- **主存储**：SQLite 单文件（`aip_obstacle.sqlite`），跟随 `--out` 目录
- **去重键**：`(airport_icao, obstacle_id, raw_text)`；`source_files.file_hash` 保证重复导入同一文件不会重新记录
- **人工修改保存**：GUI 表格中可编辑字段先只改当前确认表；点击“最终确认”后，当前确认表覆盖写入 `obstacles`，人工修改行设置 `is_user_modified=1`，GUI 中显示为蓝色
- **无用户配置文件**：MVP 所有参数走命令行或 GUI 输入框，不做持久化设置
- **无缓存层**：第一阶段不引入缓存

## 异常流

| 场景 | 处理 |
|------|------|
| 输入文件不存在 | 打印错误，退出码非 0 |
| PDF 无文本层 | 日志提示「疑似扫描件，本期不支持 OCR」，跳过 |
| PDF 加密 | 日志提示「文件加密，跳过」 |
| 未发现任何疑似障碍物行 | 日志记录「0 条候选」，正常退出 |
| 机场 ICAO 识别不到 | `airport_icao` = `UNKNOWN`，日志告警 |
| 关键字段全部缺失（无定位信息） | 进 `parse_failures`，保留 `raw_text`，流程继续 |
| 单位字段缺失 | 默认按「米」处理，并在日志告警 |
| 方位标注为磁方位（MAG） | 本期不换算，照原值存储，`parse_failures` 或日志中标警告 |
| 输出目录无权限 | 打印错误，退出码非 0 |
| 导出 GeoJSON 时经纬度为空 | 跳过该 feature，并在日志中累计「跳过数」 |

## 安全与隐私

- 中国 AIP 是公开资料，不涉及敏感个人信息
- 日志中可以记录原始行文本，无需脱敏
- 不做任何外部上传
- 不联网

---

## 与上游 / 下游的关系

- **上游**：使用者手动获取中国 AIP PDF，放到本地
- **下游**：更大的航空运行软件会读取本模块产出的 SQLite 或 GeoJSON
- **接口约定（本期不实现，仅登记）**：SQLite schema 即对外契约，修改需走兼容性评估
