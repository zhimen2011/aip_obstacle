# 需求说明

本文档仅描述 **第一阶段 MVP** 的需求边界。超出 MVP 的内容统一放在「后续阶段（不在本期范围）」一节。

---

## 1. 项目名称

AIP 障碍物数据识别与结构化模块（aip_obstacle）。

## 2. 项目目标

从 **中国民航 AIP** 中，自动识别**中国境内机场**的障碍物记录，提取关键字段，保存为结构化数据库，并支持导出为常用格式，供后续更大的航空运行软件使用。

一句话概括：把中国 AIP 中各机场的「障碍物表」从 PDF / 文本变成可被程序直接使用的结构化数据。

本模块是一个更大航空运行系统的上游数据组件，不是面向旅客或飞行员的终端产品。

## 3. 使用者

- 开发本航空运行软件的工程师（主要使用者）
- 航务、性能分析相关的业务人员（查看结果、核对数据）

使用者不一定熟悉编程，因此要提供：

- 一条命令就能跑通的入口
- 清晰的 README
- 明确的输入、输出路径和错误提示

## 4. 输入数据

MVP 只支持两种输入，且**数据来源限定为中国 AIP**（由中国民航局发布的 AIP，AD 2 章节各机场部分，其中的障碍物表通常为 AD 2.10 / AD 2.12 / AD 2.24 相关条目）：

1. **文本型 PDF AIP 文件**
   - 能直接复制出文字的 PDF（不是扫描图片）
   - 来源：中国民航局官方发布的 AIP

2. **TXT 文本文件**
   - 从 PDF 或其他来源导出的纯文本
   - 用于测试、调试、或 PDF 解析不理想时的兜底

**预留但本期不实现：**

- OCR（扫描版 PDF、图片型 AIP）
- Word / Excel 格式的 AIP
- 在线抓取 AIP
- 境外机场 AIP（格式差异大，本期不纳入）

OCR 通道会在代码中留一个空接口（`parsers/ocr_parser.py`，只抛 `NotImplementedError`），但不依赖任何 OCR 库。

## 5. 输出结果

MVP 需要产出：

1. **SQLite 数据库文件**（主存储）
   - 障碍物表 `obstacles`
   - 解析失败记录表 `parse_failures`
   - 文件导入记录表 `source_files`

2. **三种导出文件**
   - CSV：扁平表格，方便 Excel 查看。**所有记录**都导出，缺字段留空
   - JSON：结构化记录，方便程序读取。**所有记录**都导出
   - GeoJSON：带经纬度的 `FeatureCollection`。**仅经纬度齐全的记录**生成 feature；缺经纬度的记录不进 GeoJSON，但会在日志中汇总提示

3. **处理日志**
   - 纯文本日志文件，记录每次运行的文件名、识别候选条数、成功 / 失败条数、无经纬度记录数、耗时

4. **解析失败记录**
   - 识别为「疑似障碍物行」但关键字段解析失败的记录，保留 `raw_text` 和来源页码，不丢弃

## 6. 核心功能

MVP 需要实现的功能点（按数据流排列）：

1. 读取输入文件（PDF 文本层 / TXT）
2. 抽取原始文本，并保留**页码**与**所属机场 ICAO 代码**（从页眉 / 章节标题识别，例如 ZBAA / ZSPD / ZGGG 等）
3. 识别「疑似障碍物记录」——从大量文本中圈出可能是障碍物条目的行
4. 解析并结构化以下字段：

   | 字段 | 说明 | 是否必填 | 备注 |
   |------|------|--------|------|
   | `airport_icao` | 机场 ICAO 四字码 | 是 | 如 ZBAA |
   | `obstacle_id` | 障碍物编号 | 是 | AIP 原始编号，如 "1"、"OBS-001" |
   | `name` | 障碍物名称 | 是 | 如「通信塔」「烟囱」「某某大楼」 |
   | `bearing_deg` | 方位（相对机场参考点 ARP，真北方位） | 否 | 0–360 度 |
   | `distance_m` | 距离（相对机场参考点 ARP） | 否 | 统一换算为米 |
   | `latitude` | 纬度（WGS84，十进制度） | 否 | 可能缺失 |
   | `longitude` | 经度（WGS84，十进制度） | 否 | 可能缺失 |
   | `elevation_m` | 海拔高（AMSL） | 否 | 统一换算为米 |
   | `height_m` | 相对地面高（AGL） | 否 | 统一换算为米 |
   | `source_page` | 来源页码 | 是 | 溯源用 |
   | `raw_text` | 原始文本行 | 是 | 溯源用 |

   方位 / 距离 / 经纬度 / 高度可能在 AIP 里只给一部分。MVP 的原则：**给了就提取，没给就留空，不瞎猜**。
5. 写入 SQLite
6. 导出 CSV / JSON / GeoJSON
7. 输出日志和失败记录

## 7. 业务规则

MVP 版本采用**可接受即可**的规则，不追求覆盖所有历史版本格式：

- **机场归属**：通过页眉或章节标题（如 "ZBAA – BEIJING/Capital"）识别当前页所属机场；同一批导入允许包含多个机场
- **障碍物编号**：按 AIP 原文照搬为字符串，不做语义校验
- **障碍物名称**：保留原始中文 / 英文文本，不强行翻译
- **方位（bearing）**：统一为十进制度，0–360 范围；若 AIP 原文为「磁方位」需在 `parse_failures.reason` 中标注（本期**默认按真方位**处理，出现 "MAG" 字样时标警告但不换算）
- **距离（distance）**：统一换算为**米**；AIP 中出现 km / m / NM 时按单位换算，保留原始单位到 `unit_distance_original`
- **高度 / 海拔**：统一换算为**米**；AIP 中出现 ft / m 时按单位换算，保留原始单位到 `unit_height_original`
- **经纬度格式**：支持中国 AIP 常见的 `DDMMSS.ssN/S` / `DDDMMSS.ssE/W` 度分秒格式，统一转换为十进制度（WGS84）；若 AIP 不提供坐标，相关字段留空
- **记录唯一性**：以 `(airport_icao, obstacle_id, raw_text)` 作为去重键，多次导入同一份 AIP 不应产生重复记录
- **解析失败处理**：识别为「疑似障碍物行」但**关键字段（编号、名称中任一 + 至少一个定位信息：方位/距离 或 经纬度）全部缺失**时，才进 `parse_failures`；只要有编号 + 名称 + 任一定位字段，就进 `obstacles`，其他字段留空

具体的正则和识别规则会在后续开发阶段根据样例文件调整，不在本需求文档中固化。

## 8. 异常情况

必须正确处理的异常：

- 文件不存在 / 路径无权限 → 清晰错误，非 0 退出码
- PDF 是扫描件（无文本层）→ 提示「本期不支持 OCR」并跳过
- PDF 有密码 → 提示并跳过
- 文本中不含任何疑似障碍物行 → 正常结束，日志写明「本文件未发现障碍物记录」
- 机场 ICAO 代码识别不到 → 该文件仍可继续解析，`airport_icao` 记为 `UNKNOWN`，并在日志中告警
- 候选行的关键字段全部无法解析 → 进 `parse_failures`，保留 `raw_text`，流程继续
- 输出目录不存在 → 自动创建
- 输出文件已存在 → 默认覆盖，并在日志中提示

## 9. 界面要求

MVP **不做图形界面**，只做命令行：

```text
python -m aip_obstacle.cli parse <输入文件> --out <输出目录>
python -m aip_obstacle.cli export <输出目录> --format csv|json|geojson|all
```

图形界面（PySide6 / Tkinter 等）放到后续阶段。

## 10. 技术偏好

- 语言：Python 3.10+
- PDF 文本解析：`pdfplumber` 或 `pypdf`（二选一，按样例实测效果决定）
- 数据库：Python 标准库 `sqlite3`，不引入 ORM
- 导出：标准库 `csv` / `json`；GeoJSON 手写字典，不引入 geopandas
- 测试：`pytest`
- 依赖管理：`requirements.txt`（先不上 `pyproject.toml`，保持 MVP 简洁）
- 不使用：Web 框架、前后端分离、消息队列、重型 ORM、Docker（本阶段）

## 11. 验收标准

第一阶段 MVP 验收条件（全部满足才算完成）：

1. 能导入一个中国 AIP 的**文本型 PDF** 或一个 **TXT** 样例文件
2. 能从样例中提取至少一批障碍物数据（至少 ≥ 1 条真实结构化记录）
3. 每条记录能正确带上所属机场 **ICAO 代码** 和 **障碍物编号 / 名称**
4. 至少能解析「方位 + 距离」或「经纬度」其中一种定位信息
5. 能将结果写入 SQLite，且多次导入不重复
6. 能一次性导出 CSV、JSON、GeoJSON 三种文件，其中 GeoJSON 仅含有经纬度的条目
7. 解析失败的数据进入 `parse_failures` 表，`raw_text` 完整保留，不丢失
8. 普通用户按 `README.md` 从零到跑通耗时 ≤ 10 分钟
9. `tests/` 下至少有以下基础测试：
   - 经纬度格式转换（度分秒 ↔ 十进制）
   - 单位换算（ft ↔ m、km/NM ↔ m）
   - 机场 ICAO 代码识别
   - 一个端到端小样本的解析 + 导出冒烟测试
10. 所有已写测试 `pytest` 可通过
