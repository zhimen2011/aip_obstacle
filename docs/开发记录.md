## 2026-05-11（修订 2）

### 变更内容

- 根据用户反馈，**收敛数据范围为「中国境内机场」**，数据源限定为中国民航 AIP。
- **重新定义障碍物核心字段**，并同步更新三份文档：
  - 新增：`airport_icao`（机场 ICAO 代码）、`obstacle_id`（障碍物编号）、`name`（障碍物名称）、`bearing_deg`（方位）、`distance_m`（距离）
  - 保留：`latitude` / `longitude`（可选）、`elevation_m`（海拔）、`height_m`（相对地面高）
  - **移除**：`obstacle_type`、`lighting`、`marking`（本期不做，等基础管线跑通后再议）
- 更新 `docs/需求说明.md`：
  - 明确数据源为中国 AIP，不支持境外 AIP
  - 用字段表重新说明要提取的内容，并标注必填 / 可选
  - 验收标准中加入「定位信息至少提供一种（方位+距离 或 经纬度）」
- 更新 `docs/数据流说明.md`：
  - 文本抽取阶段新增「识别机场 ICAO 代码」步骤
  - SQLite `obstacles` 表结构替换为新字段集合，去重键改为 `(airport_icao, obstacle_id, raw_text)`
  - GeoJSON 导出策略明确为「仅含经纬度的条目生成 feature」
  - 新增「方位为磁方位（MAG）时本期不换算」的约定
- 更新 `docs/项目结构说明.md`：
  - 同步 `detector.py` / `field_parser.py` / `geo.py` 的职责描述
  - `tests/` 目录追加 `test_detector.py`
- 本轮**未修改** `README.md`、`AGENTS.md`、`CLAUDE.md`，也**未新增任何代码文件**。

### 变更原因

- 之前的字段集合来源于通用 AIP 障碍物描述（含灯光、标识等），但用户实际需要的是中国 AIP 中常见的「编号 / 名称 / 方位 / 距离 / 高度」五组关键信息。继续保留 `lighting` / `marking` 会让 MVP 规则过度复杂，得不偿失。
- 中国 AIP 的障碍物表中，部分条目只给「方位 + 距离」，不给经纬度；GeoJSON 的导出规则必须允许「有坐标就导出，没坐标不导出」，而不是整条记录丢掉。
- 明确数据源范围后，识别规则可以更精确（例如 ICAO 代码前缀 Z、页眉格式等），避免为「支持所有国家 AIP」付出不成比例的代价。

### 后续待办

- 等用户确认本轮修订后，再进入代码阶段。

## 2026-05-12

### 变更内容

- 调整 XLSX 导出策略：
  - 由左右双栏改为单表导出。
  - 保留解析 / 入库顺序，不再按距离重新排序。
  - 序号列显示 AIP 原始 `obstacle_id`，不再重新从 1 编号。

### 后续待办

  - 名称、方位、磁方位、距离、纬度、经度、海拔、高度可直接编辑。

  - `is_user_modified`：是否经过人工修改
  - `edited_at`：最后修改时间

- CSV / GeoJSON 导出同步带出 `is_user_modified` 和 `edited_at`。

### 变更原因

- 用户希望人工审核只处理真正需要对照 PDF 的少数记录，而不是一条条导出审核。
- 采用“界面可选修改 + 自动保存 + 导出时使用最新数据库值”的方式，可以保留原始自动解析价值，同时不强制用户补齐缺经纬度。

### 后续待办

## 2026-05-12（修订 4）

### 变更内容

- 重构 GUI 数据确认流程：
  - 解析完成后不再自动写入 SQLite。
  - 不再因为同一文件 hash 已存在而读取旧数据库记录覆盖本次解析结果。
  - 成功解析行和解析失败行一起进入“确认表”；失败行以“待补录”状态占位。
  - 新增“上方新增行 / 下方新增行 / 删除选中行 / 最终确认”操作。
  - 未点击“最终确认”前禁止导出 XLSX。
  - 点击“最终确认”后，用当前确认表覆盖同一文件 hash 的旧数据库记录。
- XLSX 导出只读取本次最终确认后的数据，避免导出上一次旧数据。
- SQLite 存储层新增覆盖保存方法，用于保存本次最终确认结果。
- 补充测试，验证失败行会回到确认表排序位置，以及同一文件 hash 的旧记录会被覆盖。

### 变更原因

- GUI 是单独工具，正确流程应是“本次解析结果优先”，旧数据库不能反向覆盖新解析数据。
- 如果 100 条记录中有 2 条解析失败，主表不能直接少掉这 2 条；需要让失败行占位并允许人工补录，保证最终 XLSX 顺序尽量与 PDF 原顺序一致。

### 后续待办

- 如果后续需要完全精确的 PDF 行序，可在解析器阶段增加显式 `source_order` 字段，而不是仅靠页码和障碍物编号排序。
## 2026-06-15

### 变更内容

- 修复 `examples/兰州_6-20.pdf` 中表格解析把部分记录误判为成功的问题。
- 新增 `services/quality.py`：
  - 表格解析为主，文本层解析作为同编号兜底。
  - 表格失败记录可用文本层结果补齐。
  - 距离明显过小、缺海拔、缺方位/距离、磁方位排序异常会降为高风险。
  - 缺经纬度不降低可信度，不触发黄/红色风险提示。
- 收紧表格方位/距离识别规则，避免把 `RWY01/36L/36R` 误识别成 `01/36`。
- PDF 解析流程改为表格解析 + 文本解析 + 质量合并后再输出 `ParseResult`。

- 兰州样例中部分记录的真正方位/距离被 pdfplumber 拆到独立文本行，表格行内只剩备注里的 `RWY01/36L/36R`。

- 用户确认中国 AIP 障碍物按 15km 内、15km 外分别按磁方位递增排序，因此增加排序质量校验。

- `python -m pytest tests/ -v`：103 passed，5 skipped。
- 兰州 PDF 当前结果：150 条成功，0 条失败；原 22 条人工修改记录的磁方位、距离、海拔均与人工 Excel 一致。

## 2026-06-30（修订 2）

### 变更内容

- 新增编号连续性质量检查：同一机场存在足够多纯数字障碍物编号时，检查最小编号到最大编号之间是否断号。
- 断号会生成 `ParseFailure`，进入 GUI 确认表的“待补录”流程，而不是只写日志。
- 缺号检查只处理纯数字编号，且同一机场至少 10 条数字编号才启用，降低误报风险。
- 新增缺号检查单元测试，并确认兰州、桂林当前样例不会产生误报。

### 变更原因

- 旧逻辑只会提示“已经成为候选行但字段解析失败”的记录；如果某条障碍物完全没进入候选行，界面不会出现待补录行。
- 编号断号检查可以在解析规则未覆盖新 PDF 抽取异常时，主动提示用户复核疑似缺失记录。

### 已知边界

- 连续性检查只能发现中间断号，例如已有 001、002、004 时提示 003。
- 如果最后一条记录也漏掉，例如当前最大编号变成 094，但实际还有 095，单靠编号连续性无法推断 095 存在；这类情况需要后续结合 PDF 原始表格行或人工标准表进一步校验。
