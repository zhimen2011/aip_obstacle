# 接手备忘录

本文档给后续接手本项目的人快速了解当前状态、运行方式和关键业务规则。先看这份，再看 `README.md`、`docs/数据流说明.md` 和 `docs/项目结构说明.md`。

---

## 一句话说明

本项目是一个中国民航 AIP 障碍物数据解析工具：从 PDF/TXT 中提取障碍物记录，允许人工复核补录，最终导出 XLSX。

当前重点不是长期数据库系统，而是一个独立工具：

```text
解析本次文件
-> 生成本次确认表
-> 人工复核 / 补录
-> 最终确认
-> 覆盖写入数据库
-> 导出本次确认后的 XLSX
```

旧数据库不能反向覆盖本次解析结果。

---

## 快速运行

安装依赖：

```bash
pip install -r requirements.txt
```

启动 GUI：

```bash
python run_gui.py
```

运行测试：

```bash
python -m pytest -q
```

如果 pytest 提示 `.pytest_cache` 无法写入，但测试全部通过，一般不影响功能。

## GUI 正确操作流程

1. 选择 AIP PDF/TXT 文件。
2. 可以先选择输出目录，也可以在导出 XLSX 前再选择。
3. 点击“解析文件”。
4. 查看“确认表”：
   - 自动解析成功的记录会直接显示。
   - 解析失败的记录会以“待补录”状态进入确认表。
   - 待补录行需要人工补齐编号、名称、方位/距离等字段。
5. 如果 PDF 中还有完全未识别出来的记录，用“上方新增行 / 下方新增行”补录。
6. 点击“最终确认”。
7. 点击“导出 XLSX”前确认输出目录正确；导出时会读取当前界面里的输出目录。

注意：未最终确认前不能导出 XLSX，这是为了避免导出上一次旧数据库数据。

如果用户在最终确认后又修改了输出目录，导出时会把本次确认数据写入新目录下的 `aip_obstacle.sqlite`，然后再生成 `obstacles.xlsx`。不要把输出目录固定在解析阶段。

## 颜色含义

| 颜色 | 含义 |
|---|---|
| 红色 | 高风险，必须人工复核 |
| 黄色 | 存在风险，需要关注 |
| 蓝色 | 已人工修改 |

蓝色优先级最高。也就是说，如果一条记录已经人工修改，即使它原本风险较高，也会显示为蓝色。

## 当前最重要的业务规则

### 1. 本次解析结果优先

GUI 解析文件后，表格显示本次 `parse_file()` 的结果。不能因为数据库里已有同一个文件 hash，就读取旧数据覆盖本次解析结果。

相关位置：

- `src/aip_obstacle/ui/main_window.py`
- `_on_parse_done()`
- `_result_rows_for_review()`

### 2. 失败行必须进入确认表占位

如果 100 条记录里有 98 条成功、2 条失败，比如 90 和 92，确认表不能只剩 98 条。失败的 90 和 92 要进入确认表，显示为“待补录”，尽量保持 PDF 原顺序。

当前排序规则主要依赖：

```text
页码 + 障碍物编号 + 解析顺序
```

如果以后要完全按 PDF 物理行顺序，建议在解析器阶段增加显式 `source_order` 字段。

相关位置：

- `src/aip_obstacle/ui/main_window.py`
- `_failure_to_review_row()`
- `_review_row_sort_key()`

### 3. 最终确认才写数据库

表格编辑、失败行补录、新增行，都先只改当前确认表。点击“最终确认”后，才覆盖写入 SQLite。

相关位置：

- `src/aip_obstacle/ui/main_window.py`
- `_on_confirm()`
- `src/aip_obstacle/storage/sqlite_store.py`
- `replace_confirmed_obstacles()`

### 4. 导出 XLSX 只能导出本次确认数据

导出前会检查：

- 是否已经解析。
- 是否已经最终确认。
- 当前会话确认的文件 hash 是否和本次解析一致。
- 数据库中本次确认数据数量是否和当前会话一致。

导出时会重新读取当前界面中的输出目录。如果当前目录还没有本次确认数据库，会先写入本次确认数据，再导出 XLSX。这样是为了保证不会导出上一次旧数据，也避免用户解析后更换输出目录时仍写到默认 `output`。

相关位置：

- `src/aip_obstacle/ui/main_window.py`
- `_on_export()`

## 主要目录职责

```text
src/aip_obstacle/
├── parsers/        # 读取 PDF/TXT，提取原始文本或表格
├── services/       # 核心解析逻辑
├── storage/        # SQLite 读写和覆盖保存
├── exporters/      # XLSX/CSV/JSON/GeoJSON 导出
├── ui/             # GUI 主窗口和人工确认流程
├── utils/          # 坐标、单位、hash、日志等工具
├── pipeline.py     # parse_file / parse_text 统一解析入口
└── models.py       # Obstacle / ParseFailure / ParseResult 等数据结构
```

不要把解析规则写进 `ui/`。GUI 只负责展示、编辑、确认和导出按钮。

## 常用代码入口

解析入口：

```python
from aip_obstacle.pipeline import parse_file

result = parse_file("examples/ZSHC.pdf")
```

核心返回值：

- `result.obstacles`：成功解析的障碍物。
- `result.failures`：解析失败、需要人工补录的记录。
- `result.stats`：统计信息。
- `result.file_hash`：文件 hash，最终确认覆盖数据库时用于定位同一文件。

最终确认保存入口：

```python
from aip_obstacle.storage.sqlite_store import SQLiteStore

with SQLiteStore("output/aip_obstacle.sqlite") as store:
    store.replace_confirmed_obstacles(
        file_path=result.source_file,
        file_hash=result.file_hash,
        obstacles=confirmed_obstacles,
    )
```

XLSX 导出入口：

```python
from aip_obstacle.exporters.excel_exporter import export_xlsx

export_xlsx(confirmed_obstacles, "output/obstacles.xlsx")
```

## 大型软件集成建议

当前项目可以作为大型软件的子模块，但建议未来再补一个稳定 API 文件，例如：

```text
src/aip_obstacle/services/api.py
```

第一版建议只暴露三个函数：

```python
parse_obstacle_file()
save_confirmed_obstacles()
export_confirmed_xlsx()
```

大型软件推荐流程：

```text
大型软件调用 parse_obstacle_file()
-> 大型软件自己的界面展示成功行和失败行
-> 用户人工确认
-> 调用 save_confirmed_obstacles()
-> 调用 export_confirmed_xlsx()
```

注意：不要让大型软件直接依赖 `ui/main_window.py`。GUI 代码是独立工具界面，不应作为长期稳定 API。

---

## 接手时最容易踩的坑

1. 不要恢复“解析完成自动写库”的旧逻辑。
2. 不要让 `file_already_imported()` 在 GUI 中决定显示旧数据。
3. 不要从数据库全量导出 XLSX；GUI 导出必须限定为本次确认后的 file hash。
4. 不要让解析失败行只停留在“解析失败”页，必须能进入确认表补录。
5. 不要把缺经纬度当作错误；缺经纬度不影响最终确认和 XLSX 导出。
6. 不要在 `ui/` 中新增复杂解析规则；解析规则应放在 `services/`。

---

## 后续优先级建议

1. 增加稳定的 `services/api.py`，方便大型软件调用。
2. 在解析器里增加显式 `source_order` 字段，进一步保证 PDF 原顺序。
3. 给 GUI 增加“只看待补录 / 只看风险 / 只看人工修改”的筛选。
4. 增加一份真实机场 PDF 的回归样例说明，方便以后验证解析质量。
