# 项目结构说明

本文档说明 **AIP 障碍物数据识别与结构化模块** 第一阶段 MVP 的目录职责与扩展约定。

> 2026-05-12 补充：当前项目已实现 `ui/` 图形界面和 `exporters/excel_exporter.py`。XLSX 导出采用单表形式，保留确认表顺序和 AIP 原始编号，不再按距离拆左右栏。GUI 表格支持人工修改、失败行补录和人工新增行；只有点击“最终确认”后才覆盖写入 SQLite 并允许导出。GUI 表格颜色含义为：红色高风险必须人工复核，黄色存在风险，蓝色已人工修改。

当前仓库里的 `src/`、`tests/`、`examples/` 仍是空目录。下面的结构是 MVP 阶段的**目标结构**，将随后续开发逐步落地。**不提前建空目录、不提前分层**。

## 各层职责（严格单一职责）

### parsers/

- **只负责**：从原始文件中拿出文本 + 页码
- **不做**：识别障碍物行、解析字段
- 每种输入一个文件，新格式新增文件，不在单文件中堆分支

### services/

- **只负责**：业务逻辑（识别、解析、标准化）
- **不做**：文件 IO、数据库读写
- 输入是 `List[TextBlock]`，输出是 `List[ObstacleRecord] + List[ParseFailure]`

### storage/

- **只负责**：SQLite 的 schema 初始化和读写
- **不做**：业务规则判断
- GUI 最终确认通过 `storage` 覆盖保存本次确认表，不让旧数据库记录反向覆盖本次解析结果
- 本期只用标准库 `sqlite3`，不引入 ORM

### exporters/

- **只负责**：把数据库里的记录写成文件
- **不做**：字段解析、单位换算（这些应已在 services 完成）
- 每种格式一个文件

### ui/

- **只负责**：文件选择、结果展示、人工修改入口和导出按钮
- 解析后生成本次确认表；解析失败行以“待补录”占位进入主表
- 表格编辑、人工补录和新增行先只影响当前确认表
- 点击“最终确认”后才调用 `storage` 覆盖写入数据库
- 未最终确认时不允许导出 XLSX
- 缺经纬度不作为颜色或审核判断依据
- 表格颜色只用于提示处理状态：红色高风险、黄色存在风险、蓝色已人工修改

### utils/

- **只放**：纯函数工具，无副作用
- 例如坐标转换、单位换算、日志初始化

### cli.py

- **只负责**：参数解析、调用各层、打印结果
- **不做**：任何业务判断或计算

## 扩展原则

1. **先有需求，再扩结构**：本 MVP 不预先创建 `ui/`、`api/`、`config/` 等目录
2. **一层对一类职责**：新功能先判断属于哪一层，不在 cli 或 parser 里写业务
3. **不引入新层**除非已存在的五层（parsers / services / storage / exporters / utils）真的放不下
4. **避免过深嵌套**：单层目录原则上不超过 2 级
5. **兼容性**：SQLite schema 变更需在 `开发记录.md` 标明，并保持向前兼容（新增列，不删列）

---

## 本期不做

以下内容 **第一阶段不建立目录、不写占位代码**（`ocr_parser.py` 是唯一例外，只抛 `NotImplementedError`）：

- `ui/`：图形界面
- `api/`：HTTP 接口
- `config/`：配置文件体系
- `scripts/`：打包脚本
- `assets/`：界面资源

待对应需求出现时再引入，避免空目录 / 空文件膨胀。
